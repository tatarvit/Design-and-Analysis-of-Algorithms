## Завдання 1 — Перевірка унікальності паролів за допомогою фільтра Блума

Реалізовано структуру даних `BloomFilter`, яка дозволяє перевіряти, чи зустрічався пароль раніше, без зберігання самих паролів.

### Що було зроблено:
- створено клас `BloomFilter` із методами `add` та `contains`;
- використано бітовий масив для компактного зберігання інформації про додані значення;
- реалізовано кілька хеш-функцій на основі стандартного модуля `hashlib`;
- додано функцію `check_password_uniqueness`, яка перевіряє список нових паролів;
- враховано некоректні значення: `None`, порожні рядки та значення, які не є рядками.

### Логіка роботи:
- пароль додається до фільтра шляхом встановлення кількох бітів у бітовому масиві;
- під час перевірки пароль вважається вже використаним, якщо всі відповідні біти вже встановлені;
- якщо хоча б один біт не встановлений — пароль вважається унікальним;
- фільтр Блума може давати хибнопозитивні результати, але не дає хибнонегативних.

### Особливості:
- паролі обробляються як звичайні рядки;
- самі паролі не зберігаються;
- структура використовує мінімум пам’яті та підходить для великих наборів даних;
- результат повертається у вигляді словника, де ключ — пароль, а значення — статус перевірки.

### Приклад результату:
```text
Пароль 'password123' — вже використаний.
Пароль 'newpassword' — унікальний.
Пароль 'admin123' — вже використаний.
Пароль 'guest' — унікальний.
```

## Завдання 2 — Порівняння продуктивності HyperLogLog із точним підрахунком унікальних елементів

Реалізовано скрипт для порівняння точного та наближеного підрахунку унікальних IP-адрес із лог-файлу `lms-stage-access.log`.

### Що було зроблено:
- реалізовано завантаження IP-адрес із лог-файлу;
- некоректні рядки та рядки без валідної IP-адреси ігноруються;
- точний підрахунок виконується за допомогою структури `set`;
- наближений підрахунок виконується за допомогою структури `HyperLogLog`;
- для обох підходів вимірюється час виконання;
- результати порівняння виводяться у вигляді таблиці.

### Логіка роботи:
- з кожного рядка лог-файлу витягується IP-адреса;
- для точного підрахунку всі унікальні IP зберігаються у множині;
- для наближеного підрахунку IP додаються до HyperLogLog;
- після обробки порівнюються кількість унікальних елементів і час виконання кожного методу.

### Особливості:
- `set` дає точний результат, але потребує більше пам’яті;
- `HyperLogLog` використовує значно менше пам’яті, але повертає наближене значення;
- рішення адаптоване для великих лог-файлів, оскільки IP-адреси обробляються послідовно;
- похибка HyperLogLog є прийнятною для задач приблизної оцінки кількості унікальних елементів.

### Приклад результату:
```text
Результати порівняння:
                       Точний підрахунок   HyperLogLog
Унікальні елементи              100000.0       99652.0
Час виконання (сек.)                0.45           0.10
```

## Завдання 1 — Оптимізація доступу до даних за допомогою LRU-кешу

Реалізовано програму, яка демонструє, як LRU-кеш пришвидшує повторні запити на обчислення суми елементів у великому масиві чисел.

### Що було зроблено:
- створено клас `LRUCache` з методами `get` та `put`;
- реалізовано функції `range_sum_no_cache` і `update_no_cache` для роботи без кешування;
- реалізовано функції `range_sum_with_cache` і `update_with_cache` для роботи з кешем;
- додано генерацію тестових запитів за допомогою функції `make_queries`;
- виконано порівняння часу роботи програми без кешу та з використанням LRU-кешу.

### Логіка роботи:
- без кешу кожен запит `Range(L, R)` обчислює суму елементів масиву напряму;
- з кешем результат для діапазону `(L, R)` спочатку шукається в `LRUCache`;
- якщо результат уже є в кеші, він повертається без повторного обчислення;
- якщо результату немає, сума обчислюється, зберігається в кеш і повертається;
- після запиту `Update(index, value)` усі кешовані діапазони, які містять змінений індекс, видаляються.

### Особливості:
- кеш має обмежену ємність `K = 1000`;
- для реалізації LRU-кешу використано `OrderedDict`;
- при переповненні кешу видаляється найдавніше використаний елемент;
- інвалідація кешу після оновлення виконується лінійним проходом по ключах кешу;
- для чесного порівняння обидва режими використовують однаковий масив і однакові запити.

### Результати тестування:
```text
Без кешу : 12.06 c
LRU-кеш  :  4.71 c  (прискорення ×2.6)
```

## Завдання 2 — Реалізація Rate Limiter з використанням Sliding Window

Реалізовано клас `SlidingWindowRateLimiter`, який обмежує частоту надсилання повідомлень користувачами в чаті.

### Що було зроблено:
- створено клас `SlidingWindowRateLimiter`;
- використано структуру `deque` для зберігання часу повідомлень кожного користувача;
- реалізовано очищення застарілих повідомлень через метод `_cleanup_window`;
- додано перевірку можливості надсилання повідомлення через `can_send_message`;
- реалізовано запис повідомлення через `record_message`;
- додано метод `time_until_next_allowed`, який показує час очікування до наступного дозволеного повідомлення.

### Логіка роботи:
- для кожного користувача зберігається історія часу його повідомлень;
- перед кожною перевіркою старі повідомлення, які вийшли за межі вікна, видаляються;
- якщо користувач ще не надсилав повідомлення або його вікно порожнє — повідомлення дозволяється;
- якщо кількість повідомлень у вікні досягла ліміту — нове повідомлення блокується;
- метод `time_until_next_allowed` обчислює, скільки секунд залишилося чекати.

### Особливості:
- використовується алгоритм Sliding Window;
- базове вікно часу — 10 секунд;
- максимальна кількість повідомлень у вікні — 1;
- при очищенні порожнього вікна запис користувача видаляється;
- результат роботи показує, чи повідомлення дозволено, або скільки секунд потрібно чекати.
